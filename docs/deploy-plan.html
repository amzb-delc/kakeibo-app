本番デプロイ手順

作成: 2026-06-15。ステータス: 未デプロイ（手順のみ）。
本番デプロイはまだ未実施。これから初回デプロイを行うための手順を自己完結でまとめる。 特に「世帯コードを本番でどう設定するか」を §4 にはっきり残す。 進行順は「仕様変更（タグ・6ヶ月グラフ・UI調整）→ Tier 8（完了） → 本デプロイ」。 本ページは spec の挙動には影響しない運用ドキュメント。

1. 構成（決定済み）

ホスティング構成 ✅ 決定

  • アプリ = Vercel（Hobby / 無料枠）。関数リージョンは東京 hnd1。
  • DB = Neon（無料枠・serverless Postgres）。
  • 無料枠のみで運用。Neon のアイドル時サスペンドは許容（§5 参照）。

2. 環境変数（Vercel に設定）

Vercel のプロジェクト設定 → Environment Variables に登録する。

変数必須内容
DATABASE_URL 必須 Neon の pooled 接続文字列（pgbouncer 経由）。アプリ実行時（サーバーレス関数）が使う。
DIRECT_URL 必須 Neon の direct 接続文字列（pooler を経由しない方）。マイグレーション / DDL 用。
作業が必要: prisma/schema.prisma の datasource に directUrl = env("DIRECT_URL") を追加する（現状未追加）。pooler 経由の接続では DDL/マイグレーションが安定しないため、Prisma の direct 接続を分離する。
SESSION_SECRET 本番必須 cookie の HMAC 署名鍵。未設定だと cookie 署名検証で 500。導入/変更すると既存 cookie が全無効化され全端末で世帯コード再保存が必要だが、本番初回なので影響なし。
ANTHROPIC_API_KEY 機能必須 レシート OCR・クレカ明細取込で必要。未設定だと該当 API が 503（家計簿の手入力自体は動く）。
OCR_MODEL / STATEMENT_MODEL 任意 OCR / 明細抽出に使うモデルの上書き。既定は OCR=haiku / 明細=sonnet。
作業が必要: OCR・明細取込の route に export const maxDuration = 60（以上）を設定する（現状未設定）。Claude への問い合わせで処理が数十秒かかり得るため、Vercel 関数の既定タイムアウトでは打ち切られる恐れがある。

3. ブートストラップ手順（ローカル or CI から Neon に向けて実行）

重要: Vercel はサーバーレスで常駐シェルが無いため、以下は「Vercel 上」ではなく 手元の PC（または CI）から本番 Neon の接続文字列を一時的に指定して実行する。本番 DSN はコミット済み .env に入れない——一時 export するか、gitignore された .env.production で扱う。
  1. Neon でプロジェクト / DB を作成 → 接続文字列を取得（pooled / direct の両方）。
  2. 本番に向けて npx prisma migrate deploy を実行（DATABASE_URL は direct 推奨）。
    本番では migrate dev 厳禁。必ず migrate deploy（既存マイグレーションを適用するだけ・schema からの差分生成やリセットをしない）。
  3. NODE_ENV=production npx prisma db seed を実行 → Tier 8 後は世帯（demo-household）＋カテゴリ 16 枠のみ作成。user 作成は廃止済み（Tier 8）。デモ支出は本番ガードでスキップされる。
  4. npm run db:set-passphrase -- "本番の世帯コード" で household.id を実コードへ付け替える（§4 ① と同一）。
  5. （任意）household.name・カテゴリ名を実名へ変更。

4. 世帯コードの設定・入力フロー

本アプリの認可は「世帯コード = household.id」方式。ユーザー管理は無く、コードを保存した端末だけがデータを見られる（spec「認証 / 世帯コード」）。本番での扱いを 2 段階に分けて明記する。

① 一度きりの設定（管理者 = あなたがローカルから実行） 設定

§3 step 4 の npm run db:set-passphrase -- "本番の世帯コード" で household.id を実コードにする。

  • db:set-passphrase は「dev 限定」ではなく DATABASE_URL 任せの DB 接続スクリプト。本番 Neon の接続文字列を指定して手元から 1 回叩く運用にする。
  • 世帯コードはアプリの UI からは変更できない（変更 UI が無い＝仕様）。変更したいときは再度このスクリプトを本番に向けて実行する。

② 各端末での入力（夫婦それぞれ・日常） 入力

  • 夫婦それぞれが端末でアプリを開き、ロック / 設定画面で ① のコードを入力する。
  • POST /api/session がコードを照合 → 一致すれば HMAC 署名付き household cookie を保存する。
  • /api/session は照合専用。コードの作成・変更はしない（アプリにコード変更 UI は無い＝仕様）。
注意（安全側の挙動）: seed 既定の demo-household は SEC-2 により本番では /api/session の保存を拒否する。つまり set-passphrase を踏まないと誰もログインできない——「設定忘れのまま誰でも入れてしまう」事故が起きない安全側の設計になっている。

5. その他の注意

  • iOS のストレージ分離: Safari とホーム画面 PWA でストレージが分離され、保存状態（世帯コードの cookie）を共有しない（MVP 許容）。Vercel が HTTPS を自動付与するため PWA の要件自体は満たす。
  • Neon の自動サスペンド: 無料枠はアイドルで自動サスペンドするため、初回アクセスが 1〜2 秒遅いことがある（許容）。
  • DDL / マイグレーションは direct 接続を使う（pooler 経由でない方）。pgbouncer 経由ではプリペアドステートメント等で問題が出やすい。
