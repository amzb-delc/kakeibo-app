Claude Code 開発ワークフロー 導入プレイブック（新規PJ用キット）

作成: 2026-06-17。ステータス: プレイブック（特定機能の設計プランではなく、新規プロジェクトに足場を組むための手順書）。
新しいプロジェクトで、本プロジェクトと同じ Claude Code 開発ワークフローの足場を最短で組むためのコピペ可能な実用キット。各項目は「プロンプト/指示の型」「設定テンプレ」「タイミング原則」の3層から成る。デプロイ先（PaaS / VPS / AWS など）に依存しない形でまとめてあり、プラットフォーム固有部分は「抽象原則＋複数例」で示す。
背景・なぜこうするのかは Claude Code 開発ワークフロー 振り返り を参照（＝振り返り＝なぜ / 本書＝どうやる の役割分担）。本書を読んで手が止まったら、対応する振り返りの節に立ち返ること。

このキットの読み方（3層）

新規プロジェクトでワークフローの足場を組むことが目的。やみくもにツールを入れるのではなく、各ステップを次の3層で捉えると、別PJでも再現できる。

プロンプト/指示の型 Claude にどう頼むか・CLAUDE.md やエージェント定義にどう書くか（＝言葉で運用する層）。
設定テンプレ ファイルとして置く設定（CLAUDE.md / エージェント定義 / フック / CI / 保護設定）。コピペの元はこのキット末尾の「テンプレ集」。
タイミング原則 いつ入れるか。とくにガードレールは「守る対象が生まれる前」に入れる。
全体を貫く一文: 保証が欲しい挙動は記憶や善意ではなく仕組み（フック / CI / ブランチ保護 / パイプライン構造）に落とす。指示の型はその仕組みを動かすための補助であって、代替ではない。

フェーズ0: プロジェクト初期化（day 0）

まだコードがほぼ無い段階。ここで「進め方そのもの」を固めると初速が出る。コードがある前から入れられる足場はすべてここで入れる。

0-1. CLAUDE.md を規約の単一の置き場にする

プロンプト/指示の型 「このプロジェクトの開発規約を CLAUDE.md にまとめて。役割分担・ファイル所有境界・ブランチ/PR運用・docs規約・コミット分割方針を含めて」。
設定テンプレ ルート直下 CLAUDE.md。最低限、次を1か所に集約する。
  • 一次情報の所在（仕様は docs/spec.html 等、CLAUDE.md は非自明点だけ、と宣言）。
  • 役割分担とファイル所有境界（誰がどのディレクトリ/ファイルを触るか）。
  • ブランチ/PR運用（feat ブランチ → PR → main、main 直コミットは軽微のみ）。
  • コミット分割方針（例: feat と docs を分ける）。
  • docs規約（自己完結HTML・起点 index・backlink・plan/backlog の置き場）。
タイミング原則 最初に置く。規約が無いまま実装を始めると、後付けの規約は既存コードと衝突して守られにくい。

0-2. サブエージェント定義（所有境界つき）

プロンプト/指示の型 「レビュー(read-only)/テスト/docs/フロント実装のサブエージェントを .claude/agents/ に定義して。各定義に『自分の領域・触ってよいファイル・push/PR まで完了させる』を明記」。
設定テンプレ .claude/agents/*.md。役割ごとに1ファイル。各 frontmatter で使えるツールを絞り、本文に所有境界とブランチ運用を書く。
役割所有境界（触ってよい範囲）書き込み
レビューなし（差分を読みコメントするだけ）read-only
テスト*.test.* 等のテストファイルあり
ドキュメントdocs/（spec/plan/backlog/index）あり
フロント実装UI コンポーネント・client state・配線あり
司令塔・バックエンド・規約残り全部・統合・PR は直列あり
タイミング原則 0-1 と同時。所有境界が決まって初めて「並行で安全に走らせられる作業」と「直列にすべき共有ハブ」を切り分けられる。

0-3. 書き込み系サブエージェントに worktree 隔離を強制するフック

プロンプト/指示の型 「.claude/settings.json の PreToolUse で、書き込み権限を持つサブエージェントが隔離ワークツリー無しで起動したら deny するフックを入れて。read-only の調査/レビューエージェントは除外」。
設定テンプレ .claude/settings.json の PreToolUse（末尾「テンプレ集」の JSON 雛形）。判定は種別で: 書き込み（Write/Edit を持つ）エージェントのみ worktree を必須化し、read-only は通す。
タイミング原則 並行作業を始める前。隔離なし起動が止まるのは事故ではなくセーフティネットで、「複数ジョブが同じ作業ディレクトリでブランチを切り替え、未コミット変更を失う」事故を構造的に防ぐ。
このフックはプロジェクト限定にする。他プロジェクトには適用しないよう、リポジトリ内の .claude/settings.json に置く（グローバル設定に入れない）。

0-4. メモリ運用方針を決める

プロンプト/指示の型 「メモリには repo/git から読めることは書かない。残すのは運用の罠・決定理由・外部システムへのポインタだけ。索引（MEMORY.md 相当）は本文を書かず一行フックに留める」。
設定テンプレ CLAUDE.md か エージェント定義に上記方針を明記。索引ファイルは常時読み込まれる前提で痩せさせる。
タイミング原則 最初に方針だけ決めておく。後から肥大化したメモリを整理するのは高コスト。

フェーズ1: 開発の足場（テストが書けるようになったら）

1-1. テスト基盤（runner ＋必要ならバージョンピン）

プロンプト/指示の型 「テストランナーを入れて最小の1本を通して。実行環境（例: Node のバージョン）と相性問題があれば、依存をピンしてその理由をメモリ/CLAUDE.md に残して」。
設定テンプレ プロジェクトのスタックに合う runner（例: 一般的な JS なら Vitest/Jest）。バージョンを固定したらその理由を必ず残す（「上げると起動失敗」等は後で必ず忘れる）。
タイミング原則 最初のロジックが書けたらすぐ。テストが無いと、後段の CI・ブランチ保護が「守る対象」を持てない。

1-2. 実装前に plan mode × 並行 read-only Explore を癖に

プロンプト/指示の型 「実装に入る前に、read-only の調査エージェントを並行で起動して、該当箇所・既存パターン・命名規則を地図化して。その上で plan mode で設計プランを出して」。
設定テンプレ 設定というより儀式。非自明な改修は「調べる → plan → 実装」を既定フローにする。Explore は読み取りのみ＝安いので隔離不要・並行可。
タイミング原則 毎回の実装の前。先に地図を持つほど plan の精度が上がり、実装後の作り直しが減る。

フェーズ2: 共有/本番の「前」に（＝守る対象ができる前に入れる）

ここがプレイブックの肝。ガードレールは「それが守る対象」より遅れて入れない。共有・本番が始まる前に CI とブランチ保護を効かせる。

2-1. CI ワークフロー（test ＋型チェック）を置く

プロンプト/指示の型 「CI を追加して。install → 型チェック → test の最小構成で。まず PR で一度走らせて緑になること、そして必須化に使うチェック名を確認したい」。
設定テンプレ CI 定義（例: GitHub Actions の .github/workflows/ci.yml。末尾「テンプレ集」に雛形）。実行環境のバージョンは 1-1 のピンと揃える。
タイミング原則 ブランチ保護の前。次項の鶏卵問題のため、まず CI を一度走らせてチェック名を実在させる。

2-2. ブランチ保護（必須チェック・PR必須）

プロンプト/指示の型 「main に PR 必須＋CI の test を必須チェックにするブランチ保護を設定して。管理者にも適用するか（enforce_admins）は方針を確認して決めて」。
設定テンプレ ホスティングのブランチ保護 API/設定（末尾「テンプレ集」に gh api 例）。必須チェック名は 2-1 で実在させたものを指定する。
タイミング原則 順序厳守 CI 設置 → 一度走らせる → 保護設定 の順。チェック名が一度も存在しないと必須化できない（鶏卵）。逆順にすると「存在しないチェックを待ち続ける」状態になる。

2-3. デプロイ手段を用意するなら、そのゲートも同時に

プロンプト/指示の型 「デプロイを用意するなら、最初からゲート済みに。test 緑を通った成果物だけが本番に出る形にして」。
設定テンプレ デプロイ経路に応じて（フェーズ3 で詳述）。最初の本番デプロイが最初からゲートされていることが重要。
タイミング原則 デプロイ導入と同時。「まず素のデプロイを通してから後でゲートを足す」は、その間に無防備な本番反映が起きうる。

フェーズ3: 本番運用（デプロイ＝プラットフォーム非依存で抽象化）

デプロイ手段は PaaS / VPS / AWS など様々。ここでは狙い（原則）を先に置き、各経路での実現例を併記する。Vercel 等の特定 PaaS はあくまで一例。

3-1. 「ドキュメント/CI設定だけの変更では本番デプロイを走らせない」

原則: 非コード変更（docs・CI 設定など）では本番デプロイを起動しない。無駄なデプロイ・無駄なリスクを避ける。実現手段はプラットフォーム依存なので、原則を満たす複数の手を持っておく。

経路実現例
CI 内 deploy ジョブdeploy ジョブに paths-ignore（docs・workflow ファイルを無視）を付ける。
自前デプロイスクリプト（VPS/AWS）スクリプト冒頭で git diff し、対象パス（例 src/）に変更が無ければデプロイを exit 0 でスキップ。
PaaS のビルド無視設定プラットフォームの「このコミットをビルドするか」判定（例: Vercel の ignoreCommand）で非コード変更を無視。
共通の狙い: 「非コード変更はデプロイしない」。手段が違っても満たすべき条件は同じ。末尾「テンプレ集」に paths-ignore と git diff 判定の2例。

3-2. テスト緑を通過した成果物だけをデプロイする（パイプライン構造で保証）

原則: 「テストをやってからデプロイする」を善意ではなく構造で保証する。一番素直なのは、CI の deploy ジョブを test ジョブに依存させる形。

経路実現例
CI（GitHub Actions 等）deploy ジョブに needs: test。test が落ちたら deploy は走らない。最も汎用（VPS=ssh 配備 / AWS=CLI 配備 / PaaS いずれにも適用可）。
VPS（ssh 配備）CI の test ジョブ成功後に deploy ジョブが ssh して配備（needs: test で連結）。
AWS（CLI 配備）同様に test 成功後の deploy ジョブで CLI 配備（needs: test）。
PaaS 固有のビルド前置プラットフォームのビルドコマンドに test を前置する手もある（例: Vercel の build スクリプトに test を挟む）。一例として軽く触れる程度に留め、基本は CI の needs: 連結を推す。
共通の狙い: デプロイ経路がどれであれ「test 緑 → デプロイ」の順序を、人の記憶ではなくパイプラインの依存関係で固定する。

3-3. 本番投入の運用テンプレと踏んだ罠を残す

プロンプト/指示の型 「本番投入の手順（マイグレーション / シード / secrets 投入）を docs に手順化して。踏んだ罠は理由つきでメモリ/docs に残して」。
設定テンプレ 本番手順は docs/<deploy>-plan.html 等に。プラットフォーム固有部分（接続文字列・secrets の入れ方・ビルドコマンド）はプロジェクト側で具体化する前提で、順序と注意点を抽象的に残す。
  • スキーマ移行は配備の前か後か（多くは「移行 → 配備」）。
  • 初期データ（seed）の要否と冪等性。
  • secrets の投入先と「鍵を変えると既存セッション/cookie が無効化される」類の副作用。
タイミング原則 初回本番投入の直後に書く（記憶が新しいうちに）。罠は再発するので「なぜ詰まったか」を必ず添える。

コピペ用テンプレ集（プラットフォーム非依存が基本）

そのまま貼って動かすのではなく、プロジェクトのスタック・ホスティングに合わせて調整する前提の雛形。固有名は一例。

A. PreToolUse「worktree 強制」フック（書き込み系のみ deny）

書き込み権限を持つサブエージェントが隔離ワークツリー無しで起動したら deny する判定の骨子。実際の判定スクリプトは「エージェント種別が書き込み系か」「現在地が隔離 worktree か」を見て、満たさなければ permissionDecision: "deny" を返す。read-only エージェントは素通し。

// .claude/settings.json（PreToolUse の骨子）
{
  "hooks": {
    "PreToolUse": [
      {
        // 書き込み系ツールの起動を判定スクリプトに通す
        "matcher": "Write|Edit|NotebookEdit",
        "hooks": [
          { "type": "command",
            "command": ".claude/hooks/require-worktree.sh" }
        ]
      }
    ]
  }
}

// .claude/hooks/require-worktree.sh（擬似ロジック）
//  1. 呼び出し元が read-only 種別（レビュー/Explore 等）なら exit 0（許可）
//  2. 現在の作業ディレクトリが隔離 worktree なら exit 0（許可）
//  3. それ以外（書き込み系がメインの作業ツリーで動こうとした）は
//     {"hookSpecificOutput":{"permissionDecision":"deny",
//       "permissionDecisionReason":"書き込み系は隔離worktreeで起動して"}} を出力

ポイントは「種別で判定」すること（書き込み系のみ必須化、read-only は除外）と、これをリポジトリ内の設定に置いて他プロジェクトに波及させないこと。

B. CI ワークフロー雛形（install → 型チェック → test）

# .github/workflows/ci.yml（GitHub Actions の最小形・一例）
name: ci
on:
  pull_request:
  push:
    branches: [ main ]
jobs:
  test:                      # ← この job 名がブランチ保護の必須チェック名になる
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20.11'   # 1-1 のピンと揃える
      - run: npm ci
      - run: npm run typecheck    # 型チェック（無ければ tsc --noEmit 等）
      - run: npm test             # テスト

スタックが JS でなければ install/型チェック/test の各行を置き換える。狙いは「PR ごとに緑/赤が出る固定のチェック名を1つ用意する」こと。

C. ブランチ保護を有効化する gh api 例

# main に「PR必須＋CIのtest必須」を設定（GitHub の一例）
gh api -X PUT repos/<owner>/<repo>/branches/main/protection \
  -H "Accept: application/vnd.github+json" \
  -f 'required_status_checks[strict]=true' \
  -f 'required_status_checks[contexts][]=test' \  # ← B の job 名と一致させる
  -f 'required_pull_request_reviews[required_approving_review_count]=0' \
  -F 'enforce_admins=true' \      # 管理者にも適用（方針次第で false）
  -F 'restrictions=null'
  • contexts[]=test は実在するチェック名（B を一度走らせて確認）でないと、永遠に保留になる。
  • enforce_admins: 管理者も保護対象にするか。小規模で緊急 push の余地を残すなら false も可だが、原則は true。
  • PR レビュー必須人数はチーム規模で。1人運用なら 0（PR は通すが直 push は禁止）でも「PR 経由＋CI 必須」は効く。

D. 「非コード変更はデプロイしない」パスフィルタ（2例）

# 例1: GitHub Actions deploy ジョブに paths-ignore
on:
  push:
    branches: [ main ]
    paths-ignore:
      - 'docs/**'
      - '**/*.md'
      - '.github/**'
# 例2: 自前デプロイスクリプト内で git diff 判定（VPS/AWS 向け）
#  直前コミットで src/ 等の対象パスに変更が無ければデプロイをスキップ
if git diff --quiet HEAD~1 HEAD -- src/ prisma/ package.json; then
  echo "コード変更なし → デプロイをスキップ"
  exit 0
fi
# ここから実際の配備（ssh / CLI など）

PaaS のビルド無視設定（例: Vercel の ignoreCommand）も同じ狙い。どの手でも条件は「非コード変更ならビルド/デプロイしない」。

E. 「test 緑のみデプロイ」例（deploy ジョブを test に依存）

# .github/workflows/ci.yml に deploy ジョブを足す（経路非依存）
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      # ... B と同じ install / typecheck / test ...

  deploy:
    needs: test                       # ← test が緑のときだけ走る
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      # VPS なら ssh 配備 / AWS なら CLI 配備 / PaaS なら deploy フック呼び出し
      - run: ./scripts/deploy.sh

PaaS 固有のビルド前置（ビルドコマンドに test を挟む）でも「test を通らないと本番に出ない」は実現できるが、経路に依存しない needs: test をまず推す。

横断原則（最終チェックリスト）

  • ガードレールは「それが守る対象」より遅れて入れない。CI はテストが書けたら、ブランチ保護は共有が始まる前、デプロイゲートはデプロイ導入と同時。
  • 保証が欲しい挙動は記憶/善意でなく仕組みに落とす。フック / CI / ブランチ保護 / パイプライン構造のいずれかで構造化する。「AI が覚えてくれる」は保証ではない。
  • 所有境界で並行、共有ハブは stack。独立した領域は並行で、複数機能が触るファイル（共有コンポーネント・スキーマ・lock ファイル等）や1回のデプロイにまとめたい作業は積み重ねる。
  • repo/git から読めることはメモリに書かない。コード構造・履歴は git log/コードが正典。メモリに残すのは運用の罠・決定理由・外部ポインタ。
  • 順序の鶏卵に注意。必須チェックは「CI 設置 → 一度走らせる → 保護設定」の順でしか組めない。
  • デプロイは経路非依存の原則で設計する。「非コード変更はデプロイしない」「test 緑のみデプロイ」を満たせれば、PaaS/VPS/AWS のどれでもよい。
