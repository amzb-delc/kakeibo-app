入力者（夫/妻）— 実装プラン

作成: 2026-06-10。ステータス: ✅ 実装完了（2026-06-10・PR #58 → #63）。 支出ごとに「どちらが登録したか（入力者）」を記録する。夫婦の2人だけなので値は 1=♂（夫）/ 2=♀（妻）。
入力者は端末ごとの設定（世帯コードと同じ cookie 方式）で持ち、新規登録時にサーバが cookie から付与する。編集では据え置き。
方針の変遷: #58 では「登録前に必須」ゲート付きで実装 → #63 で既定値＝妻(2) を起動時に発行し、必須ゲートは撤去した（下記「決定事項」参照）。一覧での入力者表示（男女アイコン）は別フェーズ。

Context（なぜ作るか）

レシートOCR・手入力・明細取り込みで支出は記録できるようになったが、夫婦どちらが登録したかは残っていない。後で「誰の支出か／誰が入れたか」を見たいというニーズがある。既存の Expense.createdByUserId は常に demo ユーザー固定の別概念（このアプリはユーザー管理を持たず、世帯コード方式）なので流用できない。そこで新フィールド enteredBy を追加する。夫の端末・妻の端末がそれぞれ自分を表すので、「入力者＝端末ごとの設定」が自然な単位になる。

既定値妻(2)。入力者 cookie が無い端末は起動時の GET /api/session で 2 を発行（妻がメイン利用の想定。夫は設定で♂に切替）。→ #63 で追加。

設計の肝

入力者を世帯コードと同じ「httpOnly cookie + SessionProvider」に乗せ、新規登録 POST 時にサーバが cookie から読んで付与する。こうすると編集 PATCH は enteredBy を一切触らないため「編集時は反映しない」が自動的に成立し、expense-form.tsx / validateExpenseInput への変更も不要になる（入力者はフォーム payload ではなく cookie 由来）。

★ 決定事項

① 入力者は「端末ごとの cookie 設定」 ✅ 決定

各配偶者が自分の端末を使うので「端末＝人」が自然。世帯コードと同じ httpOnly cookie 方式に統一。新規登録時にサーバが cookie から付与し、編集は触らない（＝据え置きが自動成立）。

② 既定値＝妻(2)・必須ゲート撤去 ✅ 決定（#63で方針転換）

#58 当初は「入力者を選ぶまで登録不可（FAB/カメラ/明細を設定へ誘導＋API 400 enteredByRequired）」だったが、運用上の摩擦を避けるため起動時に既定値 妻(2) を cookie 発行する方式へ変更。妻はそのまま登録でき、夫だけ設定で♂に切替。常に値が入るので必須ゲートは全撤去（API 400・UI 誘導・「未選択」バッジ・「登録前に必須」文言）。

トレードオフ: 夫が切替を忘れると支出が妻として記録される。妻メイン利用の前提では許容。

③ 一覧での入力者表示は別フェーズ ⏸ 後で相談

今回は「保存できる」までがスコープ。明細行への男女アイコン表示は、表示位置・既存 null データの見せ方を含めて別途設計する。

検証（実装後）

  • npx prisma migrate dev → npm run lint → npm test（入力者の付与・PATCH cookie・GET 既定値発行のテストが緑。#63 で必須ゲートのテストは撤去）。
  • npm run dev → 新しい端末（cookie 無し）で起動すると入力者＝妻、そのまま登録できる。設定で ♂ に切替 → 以降その端末の新規登録は enteredBy=1。
  • npx prisma studio で登録した支出の Expense.enteredBy を確認。その支出を編集・保存 → enteredBy が据え置き（変わらない）。
