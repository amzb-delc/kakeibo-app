家計簿アプリ — MVP仕様書

最終更新: 2026-06-10（クレカ明細PDF取り込み〈抽出→プレビュー編集→一括登録・重複候補フラグ〉。先行: UX改善・連続入力スイッチ・年月日ホイール化・OCRまとめ入力動線・明細並び替え・レシートOCR・世帯コード保護・カテゴリ管理・リファクタ Tier 5〜7）

概要

夫婦2人で日々の支出を素早く記録し、月単位で振り返るための家計簿アプリ。iPhone Safari をメインデバイスとして PWA 対応する。

項目内容
利用者夫婦2人（1世帯）
主要デバイスiPhone（Safari、PWAインストール想定）
目的支出の即時記録 + 月次の使い方の可視化
認証ユーザー管理なし。夫婦で共有する「世帯コード」＝ household.id を入力・保存した端末だけがデータを見られる方式。保存すると household cookie（httpOnly・HMAC 署名付き（SEC-3、SESSION_SECRET）・長期）を発行し、各 API は cookie 無し（未保存）・署名不正なら 401。UI 表現は「保存／この端末からクリア」（識別子・cookie名・API は unlock/lock/passphrase のまま）
タイムゾーンJST（Asia/Tokyo）固定

すべてのテーブルは世帯（Household）にスコープされる。MVP では demo-household 1件のみが seed で投入される。

Expense（支出）

カラム型備考
idString (cuid)PK
householdIdStringFK → Household（onDelete: Cascade）
categoryIdStringFK → Category（onDelete: Restrict）
amountInt円単位整数。0〜100,000,000（0円も可）

Category（カテゴリ）

カラム型備考
idString (cuid)PK
householdIdStringFK → Household
nameString世帯内ユニーク（@@unique([householdId, name])）。最大20文字
sortOrderInt表示順。世帯ごとに 0〜15 の16スロット固定
enabledBoolean@default(true)。無効カテゴリは登録/編集フォームの選択肢から除外（集計には影響しない）

16スロット固定: カテゴリは世帯ごとに 16個固定（追加・削除なし、名前変更と有効/無効の切替のみ）。プリセット名は 「ヒヨウ01〜ヒヨウ16」（slotName(sortOrder) = ヒヨウ + (sortOrder+1) を2桁ゼロ詰め。世帯内一意・非空を保証）。

必須スロット: 先頭4個（sortOrder 0..3、isRequiredSlot）は無効化できない（常に ON）。スロット定数は src/lib/category-constants.ts（prisma 非依存・副作用なしで両側から import 可）に集約。

遅延生成: 不足スロットは管理画面を開いた時（GET /api/categories?scope=all）に ensureCategorySlots() が sortOrder 0..15 の欠番を無効スロットで埋めて16枠へ収束（並行リクエストは @@unique + skipDuplicates で握り潰し）。世帯コード変更で id を付け替えた世帯でも seed を待たず揃う。

Household / User / HouseholdMember

Householdは notificationDay / notificationTime を持つが、通知機能は未実装。HouseholdMember の role は "owner" | "member" を想定。ユーザー管理は持たず、世帯共有の世帯コード（household.id）を保存した端末のみ保護対象にする。

画面構成

/ 月次サマリー（ホーム）

アプリの起点（ホーム）。開くとこの画面。月切り替え（◀▶）と、カテゴリ別の支出可視化。当月閲覧時は比較表示なし、過去月閲覧時は「今月」との差額を表示する（今月視点：今月が表示月より多ければ赤、少なければ緑）。

項目仕様
月切り替え◀▶ ボタンに加え、左右スワイプで前月/翌月。クライアント側 useState で year/month を保持し API 再取得
翌月ボタン当月なら disabled（スワイプでも当月より先には進めない）
スワイプ月送り検知レイヤー（swipeRef）を画面全体に張る。横移動が縦より優位かつ SWIPE_MIN=36px 超で発火（PWA 想定で感度高め、touchmove は passive:false）。演出は方向付きスライド＋クロスフェード: 不透明 0 の間にデータを差し替え、反対側（SLIDE_OFFSET=48px）からスライドイン
ヘッダキャラ当月表示時のみヘッダに小さなキャラ（header-character.tsx、28px、aria-hidden）。タップで一度だけ「ぴょこっ」と揺れる（animate-character-wiggle）。過去月では出さない
夫婦タグフィルタ全体カード上部に「全体 / ♂ / ♀」セグメント。選択すると GET /api/monthly-summary?tag=spouse:N を叩き、合計・ドーナツ・前月比・明細すべてサーバ側で絞られる。月送りをまたいでも選択を維持。card タグはフィルタに出さない（夫婦タグのみ絞り込み対象）。タグ仕様は 支出タグプラン
全体カードの2ペイン全体カードは横スライド式の2ペイン（切替約300ms、表示モードはカード側 state で保持）。単月ペイン＝従来のドーナツ＋レジェンド（下記）、6ヶ月比較ペイン＝直近6ヶ月のカテゴリ別積み上げ棒グラフ（下記）。単月ペインの右辺の縦長シェブロン（›）タップで6ヶ月ペインへ、6ヶ月ペインの左辺シェブロン（‹）で単月へ戻る。月送りしても表示モードは維持。夫婦タグフィルタ（上記）は両ペイン共通でカード上部。設計記録は 6ヶ月比較プラン
6ヶ月比較ペイン直近6ヶ月のカテゴリ別縦型積み上げ棒グラフ（src/components/stacked-bar-chart.tsx、自前 SVG＝recharts 等は不使用）。6本固定（古い月＝左 → 新しい月＝右、月送りで窓がスライド）、各棒はカテゴリ別に sortOrder 昇順で下から積み上げ、色は既存16色パレット categoryColor()（ドーナツと共通）。データのない月は total=0 の空棒。グラフ窓のずれ: 当月閲覧時は表示月が右端だが、過去月閲覧時は窓を未来側へ最大3ヶ月ずらす（前月＝右から2番目／前々月＝3番目／3ヶ月以上前＝右から4番目固定）。窓が実在の当月を超えることはない。これは API の sixMonths（chartKeys ベース）の並びに対応し、偏差値用の過去6ヶ月窓（sixMonthKeys）とは分離されている。正規化は6ヶ月の最大月合計を 100%。積み上げ表示は薄い色（fillOpacity 0.6）、カテゴリ単独比較時は濃い色（=1）。月ラベルは年またぎ考慮で先頭のみ年付き（例 '25 12月）。合計額ラベルは全6本に表示（省略なしのカンマ区切り・0円の月は非表示）、表示月の棒は強調（月ラベル太字＋下線・金額太字。窓ずれで右端とは限らないため selectedYm 一致判定）。目盛線はグラフ縦中央に1本（値は正規化基準＝6本中最大合計の半分・省略なし表記）。シェブロンはフローティング: ペイン内ではなく全体カードのボーダーを跨ぐ位置（左右端・縦中央・半分外側）の丸ボタンで、単月表示中は右の › のみ・6ヶ月表示中は左の ‹ のみを表示。カテゴリ単独比較: ペイン内のカテゴリ選択チップでそのカテゴリのみの単色棒6本に切替（正規化はそのカテゴリの6ヶ月最大額）、「全体」/再タップで積み上げに戻る。この選択は単月ペインのカテゴリ開閉とは独立した state。夫婦タグフィルタはサーバ側絞り込み（?tag=spouse:N）のためグラフにも反映される

API仕様

データ系エンドポイントは世帯（household cookie）にスコープされる。cookie 無し（未保存）は 401。/api/session のみ保存前でも叩ける。各ルートの定型処理（requireHouseholdId の 401 ガード・parseJsonBody・jsonError）は src/lib/api.ts に集約し、全ルートで共通化している。

GET / POST / PATCH / DELETE /api/session

世帯コードの保存・クリア・状態取得＋入力者（夫/妻）の端末保存。世帯コード = household.id、cookie は HMAC 署名付き（SEC-3、SESSION_SECRET）。コード識別子は unlock/lock/passphrase のまま（UI 表現のみ「保存／クリア」）。

  • GET — 現在の保存状態。{ unlocked: boolean, householdName?: string | null, enteredBy: 1 | 2 | null }。cookie はあるが世帯が存在しない場合は無効 cookie を破棄して返す。入力者 cookie が無ければ既定値 2=♀ を発行（妻がメイン利用の想定。夫は設定で 1=♂ に切替）
  • POST { passphrase } — 世帯コードに一致する世帯があれば household cookie（httpOnly / SameSite=lax / 約400日 / 本番のみ secure / HMAC 署名）を発行して { ok: true }。無ければ 401。同一 IP 60 秒 10 回のレート制限（SEC-1）、本番は既定世帯コードでの解錠を拒否（SEC-2）
  • PATCH { enteredBy } — 入力者を端末に保存。1/2 のみ受理し enteredBy cookie を発行（それ以外は 400）
  • DELETE — 世帯コードと入力者の cookie を両方破棄（この端末からクリア）

PATCH /api/categories/[id]

GET /api/monthly-summary

表示月のカテゴリ別集計と、比較用の「今月」集計、そして直近6ヶ月（表示月含む）の box stats（中央値・四分位・フェンス）を返す。当月閲覧時は compareTotal: null。box stats はサマリーの「異常値バー」のフィル基準として使う。各カテゴリに表示月の支出明細（spentAt 降順）を同梱するため、サマリー画面は追加通信なしで明細を展開できる。さらに全体カードの6ヶ月比較ペイン用に sixMonths（古い月→新しい月の6本固定・カテゴリ別合計）を同梱する。sixMonths はグラフ窓（chartKeys）に基づき、偏差値（box stats）用の過去6ヶ月窓（sixMonthKeys）とは分離されている: 当月閲覧時は両者一致（表示月が右端）だが、過去月閲覧時はグラフ窓を未来側へ最大3ヶ月ずらす（表示月が右から2〜4番目・実在の当月は超えない）。DB 取得範囲は両窓の和集合。route は世帯の全カテゴリを取得し、各月の byCategory の名前／色（categoryColor()）解決に使う。設計記録は 6ヶ月比較プラン。

クエリ: year, month（未指定なら当月）、tag（任意の夫婦タグ spouse:1/spouse:2 による絞り込み。isValidTag を通らない値は 400）。tag 指定時は表示月・比較月・直近6ヶ月の全クエリへ一貫適用（{ tags: { has: tag } }）＝合計・ドーナツ・前月比・明細すべてが絞られる。

POST /api/ocr

レシート画像から金額・店名・日付・カテゴリ候補を構造化抽出（Claude ビジョン）。画像は保存しない（抽出のみ）。src/lib/ocr.ts の extractReceipt() が structured outputs（JSONスキーマ）で型に一致した結果を返す。

必須環境変数: ANTHROPIC_API_KEY（未設定は 503）。モデルは OCR_MODEL（既定 claude-haiku-4-5。精度優先なら claude-opus-4-8 に上げられる）。

ボディ: { imageBase64, mediaType }（mediaType は jpeg/png/webp。画像はクライアントで長辺1568pxに縮小済み前提、上限あり）

処理: 自世帯の有効カテゴリ名を Claude に渡し、返った categoryName を完全一致で id に解決。

レスポンス: { amount: number|null, storeName: string|null, spentAt: "YYYY-MM-DD"|null, categoryId: string|null }

エラー: 残高不足/認証/混雑(429)を切り分けて 502 + { error }（クライアントに原因を表示）。

POST /api/statement

クレジットカードの利用明細PDFから利用明細行を一括で構造化抽出（Claude の document ブロック）。PDFは保存しない（抽出のみ、送信後 base64 を破棄）。src/lib/statement.ts の extractStatement() が structured outputs で cardName（明細のカード名/ブランド名。例「楽天カード」。判別不能は null・30文字以内）と rows[]（amount / spentAt / storeName / categoryName）を返す。明細は数十〜100行になるため max_tokens: 8192。

必須環境変数: ANTHROPIC_API_KEY（未設定は 503）。モデルは STATEMENT_MODEL（既定 claude-sonnet-4-6。表抽出の安定性を優先。Haiku でコスト比較も可）。

ボディ: { pdfBase64 }（上限 6MB 相当。超過は 413、欠落は 400）

処理: 自世帯の有効カテゴリ名を渡して categoryName を id に解決し、対象期間の既存支出と突合して重複候補に duplicateLikely を付与（src/lib/duplicate.ts、同日×同額×店名近似。除外はせずユーザー判断）。

レスポンス: { cardName: string | null, rows: [{ amount, spentAt, storeName, categoryId, duplicateLikely }] }。cardName はプレビューに「カード: 〇〇」として表示し、確定時に batch のトップレベル cardName として送ると全行に card:<カード名> タグが付く（支出タグプラン）。

エラー: OCR と同様に残高不足/認証/混雑(429)、および抽出側の拒否・max_tokens（明細が長すぎ）・空応答を切り分けて 502 + { error }。集計/管理行（請求額合計・繰越・手数料等）はプロンプトで除外指示。
