Tier 8 — スキーマ撤去 実装プラン

作成: 2026-06-15。ステータス: ✅ 実装完了（2026-06-15・コミット d50eb10・feat/tier8-schema-cleanup・PR 番号未定）。 リファクタ・バックログの Tier 8（旧 #12 本体）＝本番化前に残していた「未使用のマルチユーザー/通知の足場と画像予約列」を一括撤去する。
DBマイグレを伴い、かつ「完成させる or 撤去する」の設計判断が必要だったため、本番化の直前に別途実施する方針だった項目。決定は全撤去。支出データ（householdId / categoryId / tags 等）は保持する。

Context（なぜ撤去するか）

初期スキーマには「将来のマルチユーザー／通知／レシート画像保存」を見越した足場が入っていたが、MVP ではいずれも採用しないまま運用が固まった。本アプリの認可は世帯コード（household.id）の保有のみで、個人アカウントの概念は無い（spec「MVPスコープ外 / 認証・複数ユーザー」）。本番化の直前に、未使用の足場を残すか消すかを決める必要があった。

設計判断: 完成させる or 撤去する → 全撤去 ✅ 決定

  • マルチユーザー（User / HouseholdMember）は元々未使用。認可は世帯コード方式に確定しており、ユーザー単位の概念を完成させる予定がない。
  • Expense.createdByUserId は情報量ゼロ＋フットガン。常に demo ユーザー固定で入っていて分析価値が無いうえ、getDemoUserId() がデプロイ時に demo ユーザー未作成だと「demo user not found」で 500 になる地雷だった。入力者の区別は既に 夫婦タグ spouse:N（支出タグプラン）が担っている。
  • 通知（notificationDay / notificationTime）は送信ロジックが無く列だけの未実装方針。
  • 画像予約列（receiptImageUrl / ocrRawText）は「画像は保存しない」方針で書き込み経路が無い（死んだバリデーションは T5-2 で既に撤去済み）。

いずれも「いつか使うかも」の予約に過ぎず、本番化前に消す方が seed・デプロイ手順・型がシンプルになり安全。よって完成ではなく全撤去を選択。

撤去対象

対象場所理由
User モデルprisma/schema.prismaマルチユーザー未採用
HouseholdMember モデル（+ role）prisma/schema.prisma同上
Expense.createdByUserId（+ FK / relation）schema.prisma / api/expenses/route.ts / api/expenses/batch/route.ts常に demo 固定で情報量ゼロ・500 フットガン

保持するもの: Household（id=世帯コード / name / タイムスタンプ）、Category（16スロット）、Expense の支出データ本体（householdId / categoryId / amount / spentAt / storeName / memo / tags）。

マイグレーション

prisma/migrations/20260615000000_tier8_drop_user_scaffolding/migration.sql。

-- FK 解除
ALTER TABLE "Expense" DROP CONSTRAINT IF EXISTS "Expense_createdByUserId_fkey";
ALTER TABLE "HouseholdMember" DROP CONSTRAINT IF EXISTS "HouseholdMember_householdId_fkey";
ALTER TABLE "HouseholdMember" DROP CONSTRAINT IF EXISTS "HouseholdMember_userId_fkey";
-- テーブル削除
DROP TABLE "HouseholdMember";
DROP TABLE "User";
-- 通知フィールド撤去
ALTER TABLE "Household" DROP COLUMN "notificationDay", DROP COLUMN "notificationTime";
-- createdByUserId（情報量ゼロ）と予約列（画像非保存）を撤去
ALTER TABLE "Expense" DROP COLUMN "createdByUserId",
DROP COLUMN "receiptImageUrl", DROP COLUMN "ocrRawText";

支出行そのものは消えない（DROP するのは User/HouseholdMember テーブルと Expense/Household の一部列のみ）。

デプロイへの影響

  • 本番 seed が簡素化: demo ユーザーの作成が不要になり、seed から user 関連の生成を削除。set-passphrase も新スキーマに追従。
  • 「demo user not found」500 フットガンが消滅: getDemoUserId() が無くなったため、デプロイ時に demo ユーザーが無くても支出登録（POST /api/expenses / batch）が 500 にならない。bootstrap 手順から「user を作る」ステップが不要に。
  • 支出登録は createdByUserId の付与をやめた（入力者は引き続き tags の spouse:N でサーバが新規登録時のみ付与）。

検証（実装後）

  • npx prisma migrate dev → npm run lint → npm test（既存の支出登録・タグ付与テストが緑）。
  • npx prisma db seed で demo ユーザー不要のまま seed が通り、支出が再生成される（撤去前の支出データは保持される）。
  • npm run dev → 支出の新規登録・編集・明細一括取込が createdByUserId 無しで通る。npx prisma studio で User/HouseholdMember テーブルと撤去列が消えていることを確認。
