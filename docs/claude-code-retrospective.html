Claude Code 開発ワークフロー 振り返り（別PJに持ち越せる知見）

作成: 2026-06-17。ステータス: 振り返り（特定機能の設計プランではなく、横断的な読み物）。
本プロジェクト（家計簿アプリ／オーナーは開発用途で Claude Code を初めて使った）の開発を通じて得た、プロジェクトを跨いで再利用できる Claude Code 開発ワークフローの知見を抽出したもの。kakeibo 固有の仕様（画面・API・データモデル）は対象外で、それらは MVP仕様書 を参照。ここに書くのは「次のプロジェクトでも最初から効かせたい、進め方そのものの学び」。

要旨（TL;DR）

  • 足場を先に作る。役割分担サブエージェント＋worktree強制フック＋CLAUDE.md 規約は、1回テンプレ化すれば別PJにそのまま持ち込めて初速が出る。
  • 触る前に調べる。read-only の並行 Explore は安いので毎回やる。
  • Git戦略はデプロイの現実に合わせる。main マージ＝本番デプロイなら、関連PRは stack して最後に1回マージでデプロイを集約。
  • 「ハーネスで強制」か「指示で運用」かを常に意識する。保証が欲しい挙動は記憶や善意でなく仕組み（フック / CI / ブランチ保護）に落とす。これが最重要の気づき。
  • メモリ運用はスキル。repo/git から読めることは書かない。残すのは運用の罠・決定理由・外部ポインタ。

1. 並行作業の足場を最初に作ると後が速い

複数のサブエージェントに役割を持たせ、所有境界（＝誰がどのファイルを触るか）で分業を切ると、PR が小さく綺麗になり衝突が激減した。本PJでの切り方は次の通り。

領域担当（例）狙い
テスト（*.test.ts）テスト担当実装とテストの責務を分け、テスト追加でのハブ衝突を避ける
ドキュメント（docs/）docs担当spec/plan/backlog/index の更新を一手に集約
フロント実装フロント担当UI 一式（コンポーネント・state・配線）を所有
レビュー（read-only）レビュー担当コミット/PR 前の差分レビューだけを担う（書き込みなし）
司令塔・バックエンド・root・規約メイン逐次起動と統合、PR は直列

肝は「書き込み系サブエージェントの worktree 隔離を強制する PreToolUse フック」

本PJでは ファイルを書き換えうるサブエージェントを、隔離ワークツリーなしで起動しようとするとフックが弾くようにした。隔離なし起動が止まるのは事故ではなくセーフティネットであり、これにより「並行ジョブが同じ作業ディレクトリでブランチを切り替え、未コミット変更を失う／勝手に stash 退避される」事故を構造的に防げる。読み取り専用の調査エージェントやレビュアーは隔離不要なので、フックは種別で判定して書き込み系のみに worktree を必須化する。

横展開: 「役割分担サブエージェント定義」＋「worktree 強制 PreToolUse フック」＋「所有境界とブランチ運用を書いた CLAUDE.md 規約」を1回テンプレ化しておくと、次のプロジェクトに丸ごと持ち込めて初速が出る。ここは投資対効果が高い。

2. 触る前に調べる（plan mode × 並行 Explore）

実装に入る前に、read-only の Explore エージェントを並行起動して、該当箇所・既存パターン・命名規則を地図化してから着手すると手戻りが減る。「どこに似た実装があるか」「この種の関数はどう名付けられているか」を先に押さえると、設計プラン（plan mode）の精度が上がり、実装後の作り直しが目に見えて減った。

横展開: Explore は安い（読み取りのみ・並行可）。迷ったら毎回やる、を既定にしてよい。「調べてから書く」を儀式化するだけで品質が底上げされる。

3. Git/PR 戦略はデプロイの現実に合わせる

main マージ＝本番自動デプロイという構成では、関連する複数 PR を並行（parallel）ではなく積み重ね（stack）、最後に1回だけマージしてデプロイを集約すると無駄なデプロイが減る。

  • tip（先端）の PR を base=main に付け替えて squash マージ → デプロイは1回に集約。下位 PR はクローズ。
  • 共有ハブ（複数機能が同じファイルを触る箇所）は並行にすると衝突するので、最初から stack する。
原則: 「並行にできる ≠ 並行にしてよい」。独立した領域は並行、同じファイルに触れる作業や1回のデプロイにまとめたい作業は stack。判断軸は「ファイルの所有境界」と「デプロイ単位」。

4. 「ハーネスで強制」か「指示で運用」かを区別する 最重要

本PJで一番大きな気づきがこれ。lint / 型チェックが毎回走っていたのは、実はフックではなくエージェントへの指示文＋習慣だった＝構造的な保証はゼロだった。「いつも通っているから大丈夫」は、たまたま守られていただけで、忘れれば崩れる。

ハーネスで強制指示で運用
実体フック / CI / ブランチ保護CLAUDE.md・プロンプト・習慣
保証構造的（破れない）善意ベース（忘れると破れる）
例書き込み系の worktree 強制フック、PR 必須＋CI 必須チェック「コミット前に lint して」という指示
原則: 保証が欲しい挙動は、記憶や善意ではなく仕組みに落とす。「AI が覚えてくれる」は保証ではない。守られないと困るものは、フック・CI・ブランチ保護のいずれかで構造化する。

5. メモリ運用そのものがスキル

エージェントのメモリ（永続記憶）は、書く内容を選ばないと急速に陳腐化・肥大化する。本PJで効いた運用ルール。

  • repo / git から読めることは書かない。コード構造・ファイルパス・修正履歴は git log / コードが正典。
  • 残す価値があるのは「運用上の罠」「決定の理由（なぜそうしたか）」「外部システムへのポインタ」。コードを読んでも導けない文脈こそメモリ向き。
  • 「完了した＝消せる」ではない。削除の判断軸は「その情報が他のどこかに存在するか」。完了タスクでも、理由や罠が他に残っていなければ保持する。
  • インデックスは痩せさせる。MEMORY.md 的な索引は常時読み込まれるので、本文を索引に書かず一行フックに留める。
横展開: メモリは「将来の会話で役立つか」「他に正典があるか」の2軸で取捨選択する。プロジェクト状態の要約（活動ログ・アーキ図）はその時点で凍結される点に注意し、「最近の状態」は git log / コードを優先する。

6. 本番手前は実物 × レビューでゲートする

マージ前に実物で確認し、差分をレビューし、最後は CI で構造化する三段で品質を担保した。

  • 実物: dev サーバを起動して実機・実画面で挙動を確認（テストだけで満足しない）。
  • レビュー: read-only のレビュアーで差分を読む（書き込み権限を持たせないことで、レビューと実装の責務が混ざらない）。
  • 構造化: 善意の確認に頼らず、CI でチェックを必須化する（「やる」ではなく「通らないとマージできない」へ）。
原則: 「テスト緑」と「実際に動く」は別物。手元の実機確認を省かない。そして人手のレビュー・確認は、最終的に CI へ吸い上げて強制に変える（4. と地続き）。

7. 詰まり所（次回先回りできる）

症状正体対処
作業ツリーが別ブランチの中身に戻る並行作業中のブランチ切替（多くは worktree 未隔離が原因）慌てない。成果はブランチ／PR が持つと切り分ければ失っていない。再発防止は worktree 隔離の強制（1.）。
2か所の値がいつの間にかズレる手動同期ポイント（例: package.json の版番号と、UI 表示用の版番号定数）同期点を減らす（単一の真実源にする）か、ズレたら気づける形にする（テスト/CI で突き合わせ）。手動同期は漏れる前提で設計する。
使い捨て worktree が溜まる隔離起動の副産物が掃除されず蓄積定期的に掃除する運用を入れる。worktree はあくまで使い捨て＝成果の永続性はブランチ側にある、と割り切る。

まとめ：次のPJに持ち込むチェックリスト

  • 役割分担サブエージェント定義＋worktree 強制フック＋CLAUDE.md 規約をテンプレ化して持ち込む。
  • 実装前に並行 Explore で地図化する習慣を既定に。
  • デプロイ単位に合わせて PR を stack か parallel かを意識的に選ぶ。
  • 守られないと困る挙動は、指示ではなくハーネス（フック / CI / ブランチ保護）で強制する。
  • メモリは「他に正典があるか」「将来役立つか」で選別し、索引は痩せさせる。
  • マージ前は実物確認 → read-only レビュー → CI 必須化の三段でゲートする。
