クレカ明細PDF取り込み — 実装プラン

作成: 2026-06-09。ステータス: ✅ 実装完了（2026-06-10・PR #48）。実際の入口はホームのヘッダ左「フォルダ」アイコン（設計途中で設定モーダル内から変更）、それ以外は本プラン通り。詳細は MVP仕様書 参照。以下は設計時の記録。
レシートOCRパイプライン（lib/ocr.ts）を踏襲し、カード明細PDFをClaudeで構造化抽出 → プレビュー編集 → 一括登録する。CLAUDE.md で Phase2 に後回しだった機能。
3論点 確定済み（2026-06-09）: 入口=設定モーダル内＋ピッカー自動オープン / 返金マイナス行=取込OFF既定で除外 / 一括登録=全行ロールバック＋プレビュー行トグル。末尾「決定事項」参照。実装に入れる状態。

Context（なぜ作るか）

日々の支出はレシートOCR・手入力でカバーできているが、クレジットカード払いは明細が後からまとめて来るため取りこぼしやすい。月次でカード明細PDFを丸ごと取り込めれば、記録漏れが大幅に減る。当初は「iPhoneでのCSV/PDF取り回しが課題」で Phase2 送りだったが、ただのファイルピッカー（<input type="file" accept="application/pdf">）で iOS Safari / PWA から明細PDFを選べることが確認できたため着手する（iOS は Web Share Target 非対応なので共有シート連携はしない）。

入口の置き場所

現状フッタは 左=設定 / 中央=FAB / 右=レシートOCRカメラで3スロット埋まっている。一方、ホームのヘッダは左スロットが空いている（ホームは戻るボタンが無いため backHref 無し＝左 w-11 が空き、page-header.tsx:31-41）。ここを使う。

✅ 決定 ホームのヘッダ左スロットに「フォルダ」アイコン（page-header.tsx に left? プロップ追加）。タップでファイルピッカー → プレビュー。未保存（unlocked でない）ときは非表示（FAB/カメラと同じガード）。右スロットの HeaderCharacter と左右バランスも取れる。

iOS制約メモ: 「指定フォルダを自動で読む」（File System Access API）は iOS Safari / PWA 非対応のため不可。iOSの手数下限は「ピッカーで1タップ選択」。将来の自動取り込み（iOSショートカット→/api/statement POST、またはクラウドフォルダのバックエンド監視）は /api/statement を再利用できる形にしておき、別フェーズ・ホスティング決定後に検討。

structured output schema（lib/statement.ts 内）

structured outputs は配列直返しよりオブジェクト直下に配列が安定（全プロパティ required ＋ additionalProperties:false を満たすため）。minItems/maxItems 等の配列制約は非対応なので、行数はプロンプト指示＋サーバ側チェックで担保。

{ type: "object", additionalProperties: false,
  properties: { rows: { type: "array", items: {
    type: "object", additionalProperties: false,
    properties: {
      amount:       { type: ["integer","null"], description: "利用金額(円)。返金はマイナス。読めなければnull" },
      spentAt:      { type: ["string","null"],  description: "利用日 YYYY-MM-DD" },
      storeName:    { type: ["string","null"],  description: "利用店名・加盟店名" },
      categoryName: { type: ["string","null"],  description: "与えたカテゴリ一覧から完全一致で1つ。なければnull" },
    },
    required: ["amount","spentAt","storeName","categoryName"],
  } } },
  required: ["rows"] }

ルート側で categoryName → categoryId を解決（api/ocr/route.ts:57-59 と同じ categories.find ループ）。

Anthropic PDF document ブロックの渡し方

SDK @anthropic-ai/sdk v0.100.1。extractReceipt（ocr.ts:73-94）の image ブロックを document ブロックに差し替えるだけ。document(PDF) と output_config.format（json_schema）の併用は可（PDFは入力側のvision、schemaは出力側の制約で独立）。

const res = await client.messages.create({
  model: STATEMENT_MODEL,
  max_tokens: 8192,               // 明細は数十〜100行。レシートの1024では足りない
  messages: [{ role: "user", content: [
    { type: "document",
      source: { type: "base64", media_type: "application/pdf", data: pdfBase64 } },
    { type: "text", text: instruction },   // カテゴリ名リスト注入は extractReceipt と同様
  ]}],
  output_config: { format: { type: "json_schema", schema: STATEMENT_SCHEMA } },
});
  • 応答処理は extractReceipt と同一（refusal→拒否、text→JSON.parse）。追加で stop_reason==="max_tokens" を検出し「明細が長く全件読めなかった可能性」エラーにする。

モデル選定

レシートOCRは claude-haiku-4-5。明細は長い表＋多数行の構造化抽出で取りこぼし・桁ずれが起きやすい。推奨: STATEMENT_MODEL の既定を claude-sonnet-4-6（structured outputs対応・表抽出が安定・月数回の低頻度なのでコスト影響小）。env で切替可（process.env.STATEMENT_MODEL ?? "claude-sonnet-4-6"）にして Haiku と実データ比較も可能に。Opus 4.8 は過剰。

サイズ / ページ上限 ⏸ 実装前に実機確認

  • PDF base64 直渡しの公称上限はおおむね 100ページ / 32MB（リクエスト全体）。正確な値は platform.claude.com/docs/en/build-with-claude/pdf-support.md で確認（今回ライブ取得できず）。クレカ明細は通常数ページなので実用上は問題になりにくい。
  • デプロイ環境（ホスティング未決）のリクエストボディ上限に注意。超える場合は Files API 経由（files.upload → document.source.{type:"file", file_id}）へ切替余地を残す。初版は base64 直渡し＋控えめ上限（例 base64 6MB）。

重複検出

サーバ側で判定（抽出API内、findDuplicateFlags）。クライアントに既存支出を渡さず、世帯スコープのDBクエリで安全に突合。

  • 抽出行の有効日付（parseReceiptDate）の min/max から対象期間を date.ts のJST境界ユーティリティで算出 → その期間の既存支出を1クエリ取得（amount/spentAt/storeName）。
  • 「重複かも」= 同日 かつ 金額一致 かつ 店名が近い（店名は trim/小文字化して包含 or 一致。店名null行は日付×金額のみの弱一致）。
  • 判定本体は純関数 markDuplicates(rows, existing) に分離して prisma 非依存にしテスト容易化。除外はしない（取込トグル既定ON、バッジ表示のみ）。

エッジケース

ケース対応
長い明細（数十〜100行）max_tokens: 8192（必要なら16384）。件数が極端（>300等）ならサーバで警告。
複数ページPDFdocument ブロックがそのまま受ける。上限超は 413。
抽出失敗・部分失敗max_tokens→専用エラー、refusal→拒否、JSON.parse失敗→502、rows空→「読み取れませんでした」。
返金・キャンセル（マイナス）schema で負を許容して抽出。登録可否は要決定②。
合計行/手数料行の混入プロンプトで「合計・繰越・お支払金額・手数料・ポイント等の集計行は除外、個別利用明細のみ」と指示。最終防衛線はプレビューで取込OFF。
PDFサイズ上限クライアントで file.size 事前チェック、サーバでも 413。
パスワード付きPDF要件外（なし前提）。開けないPDFは「対応していないPDFです」。

★ 決定事項（2026-06-09 確定）

① 入口の置き場所 ✅ 決定

ホームのヘッダ左スロットにフォルダアイコン（page-header.tsx に left? 追加、ホームは backHref 無しで左が空き）。タップでファイルピッカー→プレビュー。unlocked 時のみ表示。フッタは増やさない。

「指定フォルダ自動読み込み」は iOS（File System Access API 非対応）で不可のため見送り。将来の自動化（iOSショートカット→API / クラウド監視）は /api/statement 再利用で別フェーズ・ホスティング決定後。

② 返金・キャンセル行（マイナス金額） ✅ 決定

取込OFF既定で除外。既存 validateExpenseInput（amount >= 0）を変えず、プレビューで「返金」表示・OFF（手動ONも可）。集計（ドーナツ/異常値バー）への影響を初版で抱えない。

③ プレビュー＆一括登録 ✅ 決定

全行をプレビューに読み込み、登録する行を行トグルで選ぶ（取込OFF行は送信しない）。バリデーション失敗時は全行ロールバック（$transaction・何も作らず errors 返却）。プレビューで弾けるので実害は小さい。
