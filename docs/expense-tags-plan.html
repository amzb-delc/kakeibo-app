支出タグ（enteredBy → tags）— 実装プラン

作成: 2026-06-12。ステータス: ✅ 実装完了（2026-06-12・PR 番号未定 / feat/expense-tags）。 支出ごとの Expense.enteredBy Int?（入力者 1=♂/2=♀）を廃止し、汎用的な内部タグ配列 Expense.tags String[] へ一般化する。
入力者は 夫婦タグ spouse:1(♂)/spouse:2(♀) として、クレカ明細のカードは カードタグ card:<カード名> として記録する。タグは内部仕様（画面に #xxx 形式は出さない）。
付与・据え置きの方針は 入力者プランを踏襲（新規登録時にサーバが付与・編集 PATCH は据え置き）。設定の夫/妻選択・cookie・/api/session は従来どおり存続。新たに明細行の識別ドットと全体カードの夫婦タグフィルタを追加。

Context（なぜ tags へ一般化するか）

入力者プランで導入した Expense.enteredBy Int? は「夫/妻のどちらが登録したか」専用の単一列だった。その後、クレカ明細取込でどのカード由来の支出かも残したくなったが、列を増やすたびにスキーマ・API・フォームに分岐が増える。そこで「支出に複数の機械可読ラベルを付けられる」汎用の tags String[] に一般化し、入力者もカードもこの上に乗せる。

タグは内部仕様で、画面に #xxx 形式のチップは出さない。種別は名前空間プレフィックス（spouse: / card:）で機械判定し、UI では識別ドットや絞り込みなど用途ごとの表現に変換する。これにより「タグ機能のUIを作り込む」コストを掛けずに、データだけ先に汎用化できる。

タグの内部形式（src/lib/tags.ts が単一ソース）

タグの形式・検証・色付けはすべて src/lib/tags.ts に集約する（コード/メモリに散らさない）。

種別形式由来制約
夫婦タグspouse:1(♂) / spouse:2(♀)設定の入力者 cookie（getEnteredBy()）2値のみ（spouseTagOf(1|2)）。isSpouseTag で判定
カードタグcard:<カード名>クレカ明細抽出の cardNametrim 後 1〜30文字（MAX_CARD_NAME_LENGTH）・制御文字(C0/DEL)不可。空・超過・不正は null＝タグ化しない（cardTagOf）
  • isValidTag(tag) = isSpouseTag || isCardTag。isCardTag は cardTagOf との往復一致で「trim 済み・上限内・制御文字なし」を保証（不正タグの混入を弾く）。
  • tagColor(tag) = 識別ドットの色（hex）。spouse は固定色（spouse:1=青 #3b82f6 / spouse:2=ローズ #f43f5e）、card はカード名の djb2 ハッシュで6色パレットから安定選択（同じカード名は常に同じ色）。

記録タイミング（入力者プランを踏襲）

新規登録（FAB / レシート / 明細一括）
  POST /api/expenses ───────────────▶ tags = [ spouseTagOf(getEnteredBy()) ]      ← cookie 由来
  POST /api/expenses/batch ─────────▶ tags = [ spouseTagOf(...), card:<cardName> ]  ← cookie + リクエストの cardName

編集
  PATCH /api/expenses/[id] ─────────▶ tags は data に含めない＝据え置き（enteredBy 時代と同じ）

cookie・設定モーダルの夫/妻選択・PATCH /api/session・既定値2(♀)発行はすべて従来どおり存続。違いは「cookie の値を enteredBy 列ではなく tags の spouse:N として書く」点だけ。

データモデル / マイグレーション

model Expense {
  …
  tags  String[]  @default([])   // 内部タグ。spouse:1|2 / card:<名前>
}

マイグレーション 20260612051542_replace_entered_by_with_tags はロスレス移行:

-- 1) tags 列を追加
ALTER TABLE "Expense" ADD COLUMN "tags" TEXT[] NOT NULL DEFAULT ARRAY[]::TEXT[];
-- 2) 既存 enteredBy をバックフィル（1→spouse:1 / 2→spouse:2、NULL はタグなし）
UPDATE "Expense" SET "tags" = ARRAY['spouse:' || "enteredBy"] WHERE "enteredBy" IS NOT NULL;
-- 3) enteredBy 列を削除
ALTER TABLE "Expense" DROP COLUMN "enteredBy";

API 変更

エンドポイント変更
POST /api/expensescookie の入力者を spouse:N タグとして create に付与（ボディでは受け取らない）。
POST /api/expenses/batchトップレベルに cardName?: string を受理。各行に spouse:N ＋（cardTagOf(cardName) が非 null なら）card:<名前> を付与。cardName が string 以外は 400。
POST /api/statement抽出スキーマに cardName を追加（カード名/ブランド名を30文字以内で。判別不能は null）。レスポンスに cardName: string | null を追加。
GET /api/monthly-summaryクエリ ?tag= を追加。isValidTag を通らない値は 400。指定時は表示月・比較月・直近6ヶ月の全クエリへ { tags: { has: tag } } を一貫適用（合計・ドーナツ・前月比・明細すべてサーバ側で絞る）。各明細行に tags: string[] を同梱。
PATCH /api/expenses/[id]変更なし（tags を扱わない＝据え置き）。

UI（識別ドット・夫婦フィルタ）

  • 明細行の識別ドット: 各支出の店名/メモの前に、その行の tags ごとに tagColor(tag) の小さな丸ドットを並べる （spouse:1=青 / spouse:2=ローズ / card= カード名ハッシュで6色パレットから安定選択）。これにより「一覧での入力者表示」が解消（入力者プランで別フェーズだった項目）。
  • 全体カードの夫婦タグフィルタ: 全体カード上部に「全体 / ♂ / ♀」セグメントを置き、選択で GET /api/monthly-summary?tag=spouse:N を叩く。合計・ドーナツ・前月比・明細がすべて絞られる。月送りをまたいでも選択を維持。card タグはフィルタに出さない（夫婦タグのみ絞り込み対象）。

明細取込のカード名フロー

PDF → /api/statement
   extractStatement() が cardName + rows[] を構造化抽出
   ─▶ レスポンス { cardName: "楽天カード" | null, rows: [...] }

プレビュー画面
   「カード: 〇〇」を表示（cardName が null なら出さない）

★ 決定事項

① 専用列ではなく汎用 tags String[] に一般化 ✅ 決定

入力者もカードも「支出に付く機械可読ラベル」という同じ形。列を増やすたびにスキーマ/API/フォームへ分岐が波及するのを避け、配列1本に集約。種別は名前空間プレフィックスで判定する。

② タグは内部仕様（#xxx チップを出さない） ✅ 決定

ユーザーにタグを編集させる汎用UIは作らず、用途ごとの表現（識別ドット・夫婦フィルタ・カード名表示）に変換する。フリー入力のタグ機能は今回スコープ外。

③ 付与・据え置きは入力者プランを踏襲 ✅ 決定

新規登録時にサーバが cookie/リクエストからタグを付け、編集 PATCH は据え置き。expense-form / validateExpenseInput には乗せない（タグは payload 由来でない）。

検証（実装後）

  • npx prisma migrate dev → npm run lint → npm test（タグ付与・cardTagOf/isValidTag の境界・?tag= 絞り込みと不正タグ 400・据え置きのテストが緑）。
  • npm run dev → 端末の入力者を♂/♀で切替 → 登録した支出の明細行に対応色のドットが出る。全体カードの「♂/♀」で合計・ドーナツ・明細が絞られ、月送りでも維持される。
  • クレカ明細PDFを取込 → プレビューに「カード: 〇〇」 → 一括登録後、各行に card:〇〇 タグ（ドット）が付く。npx prisma studio で Expense.tags を確認。編集・保存しても tags は据え置き。
